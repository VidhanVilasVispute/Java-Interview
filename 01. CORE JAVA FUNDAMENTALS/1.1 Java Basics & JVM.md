## 1. What is Java and what are its key features?

Java is a **statically typed, object-oriented** programming language that compiles source code into platform-independent **bytecode**, which runs on the JVM.

Key features include:

- Platform independence through the JVM
- Automatic memory management via garbage collection
- Strong multithreading support with a well-defined memory model
- Rich standard libraries
- Built-in security mechanisms
- High performance via JIT (Just-In-Time) compilation

## 2. Is Java 100% object-oriented? What are the advantages of Java being partially object-oriented?

No, Java is not 100% object-oriented because it has primitive data types like `int`, `char`, `boolean`, and `double` that are not objects. This design improves performance and memory efficiency since primitives are stored directly and processed faster than objects. Whenever object behavior is required, Java provides wrapper classes such as `Integer`, `Double`, and `Character`, along with autoboxing and unboxing.

## 3. What is the difference between primitive and non-primitive data types? Can primitive data types be null?

Primitive data types are built-in types like int, char, and boolean that store values directly. They are faster, consume less memory, and do not have methods. Non-primitive data types are reference types such as String, arrays, classes, and wrapper classes. They store references to objects, support methods, and can be assigned null. Primitive data types cannot be null because they store actual values rather than object references.

## 4. Can we declare pointers in Java?

No, Java does not support explicit pointers. Instead, it uses reference variables to access objects. References do not expose memory addresses or allow pointer arithmetic. This improves security, prevents memory corruption, simplifies programming, and enables automatic memory management through the Garbage Collector.


## 5. What is the difference between JVM, JRE, and JDK?

JVM is the Java Virtual Machine that executes Java bytecode. JRE is the Java Runtime Environment that includes the JVM and the libraries required to run Java applications. JDK is the Java Development Kit that includes the JRE along with development tools like javac, jdb, and jar for building, debugging, and packaging Java applications. In short, JDK is for development, JRE is for execution, and JVM is the engine that runs the bytecode.


## 6. What are the key components of JVM Architecture?

The JVM architecture consists of the Class Loader, which loads class files; the Bytecode Verifier, which validates bytecode for security; the Runtime Data Areas, including the Heap, Stack, Method Area, PC Register, and Native Method Stack; the Execution Engine, which executes bytecode using the Interpreter and JIT Compiler; the Garbage Collector, which automatically reclaims unused memory; and the JNI with Native Libraries, which allow Java to interact with platform-specific code.


## 7. Can a Java application be run without installing the JRE?

In older Java versions, a separate JRE was required to run Java applications. In modern Java (11 and later), Oracle no longer provides a separate JRE because the JDK already includes the runtime components. So, if the JDK is installed, a separate JRE is not needed.

## 8. Is it possible to have JDK without JRE?

No. The JDK cannot exist without the Java runtime because it needs the JVM and Java libraries to compile and execute programs. While modern Java versions don't ship a separate JRE package, the runtime components are still included in the JDK.

## 9. What makes Java "write once, run anywhere"? What is the Java Virtual Machine (JVM) and how does it work?

Java is called "Write Once, Run Anywhere" because Java source code is compiled into platform-independent bytecode. This bytecode can run on any operating system that has a compatible JVM. The JVM loads the class files, verifies the bytecode, manages memory, executes the code using the Interpreter and JIT Compiler, and converts the bytecode into native machine code for the underlying operating system. This allows the same Java program to run on different platforms without recompilation.

## 11. What are the types of memory areas allocated by JVM? Which is faster to access between heap and stack?

The JVM memory is divided into five main areas: Heap, Java Stack, Metaspace, PC Register, and Native Method Stack. The Heap is shared among all threads and stores objects and arrays. The Stack is thread-specific and stores local variables, method parameters, and stack frames. Metaspace stores class metadata, while the PC Register tracks the current instruction, and the Native Method Stack supports JNI calls. Between the Heap and the Stack, the Stack is faster because it uses a simple LIFO structure, requires no garbage collection, and supports efficient memory allocation and deallocation.

## 12. What is the role of the JIT (Just-In-Time) compiler?

The JIT (Just-In-Time) Compiler is a part of the JVM's Execution Engine. It improves performance by identifying frequently executed bytecode, called hot code, and compiling it into native machine code. This avoids interpreting the same code repeatedly, resulting in faster execution. The JIT also performs runtime optimizations, making long-running Java applications more efficient.

## 13. What is a classloader in Java? How does class loading affect memory usage?

A ClassLoader is responsible for loading Java `.class` files into the JVM at runtime. Java uses three main ClassLoaders: Bootstrap, Platform (Extension), and Application, following a parent-first delegation model to improve security and avoid duplicate class loading. The class loading process has three phases: Loading, Linking (Verification, Preparation, Resolution), and Initialization. Class metadata is stored in Metaspace, so loading more classes increases Metaspace usage. If too many classes are loaded and cannot be unloaded, it may result in an `OutOfMemoryError`

## 14. Is it possible to unload a class in Java?

Yes, class unloading is possible, but not directly. A class is unloaded only when the ClassLoader that loaded it becomes unreachable and is garbage collected. Additionally, there must be no live instances of the class and no references to its Class object. Class unloading helps free Metaspace memory.

## 15. Which version of Java are you using?

In production, use Long-Term Support (LTS) versions such as Java 17 or Java 21. These provide stability, long-term security updates, and strong ecosystem support. Non-LTS versions have shorter support cycles and are not preferred for enterprise applications.
